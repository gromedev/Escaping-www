Azure AD Conditional Access Troubleshooting

When Conditional Access policies cause unexpected sign-in failures, use this systematic troubleshooting guide to identify and resolve the issue.

Step 1: Identify the Blocking Policy

Sign-in Logs: Navigate to Azure AD > Sign-in logs > select the failed sign-in > Conditional Access tab. This shows which policies evaluated and their result (Success, Failure, Not Applied).

What-If tool: Azure AD > Security > Conditional Access > What If. Simulate a sign-in to see which policies would apply.

Step 2: Common Failure Patterns

Pattern A: MFA Required but Not Registered

Symptom: User sees "Your organization requires additional verification" but can't complete MFA setup.

Fix: Ensure the Combined Registration policy allows users to register at aka.ms/setupsecurityinfo. Check that the registration CA policy doesn't also require MFA (circular dependency).

Pattern B: Compliant Device Required but Shows Non-Compliant

Symptom: Device is enrolled in Intune and compliant, but CA still blocks.

Fix: Check the Primary Refresh Token (PRT) status: dsregcmd /status. Look for AzureAdPrtUpdateTime. If stale, run dsregcmd /refreshprt. Also verify the device object in Azure AD matches the Intune device object (same deviceId).

Pattern C: Location-Based Block

Symptom: User blocked when connecting from a specific network or VPN.

Fix: Check Named Locations in Azure AD > Security > Conditional Access > Named locations. Verify the IP ranges match your egress IPs. Remember that IPv6 addresses may differ from IPv4 named locations.

Pattern D: App-Specific Block

Symptom: User can access some apps but not others.

Fix: Check which cloud apps are targeted by the blocking policy. Some apps (like Exchange Online) have separate service dependencies. Verify that Office 365 Exchange Online is included if Outlook is blocked.

Step 3: Emergency Procedures

If all users are locked out: Use an emergency access account (break-glass) that's excluded from all CA policies. Navigate to Azure AD > Security > Conditional Access and set the blocking policy to Report-only mode while investigating.

Step 4: Validate the Fix

  1. Use the What-If tool to confirm the policy change produces the expected result
  2. Set the policy to Report-only mode first to observe impact without enforcement
  3. Monitor sign-in logs for 24 hours before switching to On
  4. Verify with the affected user(s) that access is restored

Useful PowerShell Commands

# Get all CA policies
Get-MgIdentityConditionalAccessPolicy | select DisplayName, State

# Check user's group memberships (for policy targeting)
Get-MgUserMemberOf -UserId user@domain.com | select -ExpandProperty AdditionalProperties

# Check device compliance status
Get-MgDeviceManagementManagedDevice -Filter "userPrincipalName eq 'user@domain.com'" | select DeviceName, ComplianceState
